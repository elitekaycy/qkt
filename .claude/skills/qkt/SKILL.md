# qkt — project conventions

This skill is the source of truth for **how** we work in the qkt repository. The codebase is a long-horizon, event-driven trading engine in Kotlin that will eventually expose a SQL-like DSL for trading strategies. We treat the repository as a future open-source project from day one — meaning every commit, file, and process should be defensible to a stranger reading it cold.

**This document is living.** It grows as the project grows. When a convention is missing or wrong, fix the skill in the same PR as the code change that motivated it.

---

## 0. Mission and audience

- **Mission.** Build a deterministic, testable, event-driven trading runtime in Kotlin that can backtest historical data with bit-identical results to live execution, and that compiles a SQL-like DSL into runnable strategies.
- **Audience.** Future contributors (including future-you), both engineers and quants. Code should be readable cold. Specs and plans are first-class artifacts; they live in `docs/superpowers/`.
- **Open-source posture.** No private references in commits, code, or comments. No AI references anywhere — no `Co-Authored-By`, no "generated by", no `// AI:` markers. The history must look like the work of a small team of disciplined humans.

---

## 1. Phases

Trading-engine work is structured in numbered phases. Each phase has its own spec and plan.

| Phase | Scope | Status |
|---|---|---|
| 1 | Core engine MVP — tick → strategy → signal → order → broker → trade | shipped |
| 2 | Event bus, candle aggregator, multi-strategy support, SLF4J | planned |
| 3 | Risk engine, position tracking, P&L, `BigDecimal` for prices | planned |
| 4 | Backtest replay engine — deterministic execution against historical data | planned |
| 5 | DSL — internal Kotlin DSL first, then external SQL-like parser | planned |

Every PR title and body must reference the phase it contributes to (see PR template below). Cross-phase work is not normally permitted; if a Phase 2 PR touches Phase 1 code, justify it in the PR description.

## 2. Branching strategy

qkt uses a three-branch promotion pipeline. Changes flow one way only:
`feature → dev → testing → main`.

- **`dev`** is the integration branch and the GitHub default branch. All work
  lands here first. It runs the fast essentials CI (compile + ktlint + unit
  tests).
- **`testing`** is the staging branch. It is updated *only* by automatic
  fast-forward promotion from `dev` once `dev` passes the essentials CI. It runs
  the full black-box integration CI.
- **`main`** is the release-ready branch. It is updated *only* by *manual*
  fast-forward promotion from `testing`, after `testing` passes the integration
  CI. Tags and releases are cut from `main`.

Never commit directly to `testing` or `main` — they are promotion-only. The
promotion is a fast-forward, so it creates no merge commit.

- `main`, `testing`, and `dev` must always build and test green.
- Feature branches branch from `dev` and merge back into `dev`. They are named
  after their phase and feature: `phase<N>-<short-feature-name>`. Examples:
  `phase2-event-bus`, `phase5-dsl-parser`.
- Bugfix branches: `fix-<short-description>` (no phase prefix; bugs aren't
  phase-scoped). Refactor branches: `refactor-<short-description>`.
- One concern per branch. Don't pile two features into one branch.
- Never commit directly to `dev`. Always go through a feature branch + PR.

The pipeline is specified in
`docs/superpowers/specs/2026-05-22-ci-promotion-pipeline-design.md`.

## 3. Commit conventions

We use **Conventional Commits** with strict project rules.

### Format

```
<type>(<scope>): <subject>
```

Subject only. **No body. No footer. No `Co-Authored-By` lines. No AI references of any kind.**

### Allowed types

| Type | Use for |
|---|---|
| `feat` | New user-visible behavior, new types, new components |
| `fix` | Bug fix in existing behavior |
| `refactor` | Internal restructuring with no behavior change |
| `docs` | Documentation, specs, plans, README, comments |
| `test` | Adding or updating tests without changing production code |
| `style` | Formatting-only changes (whitespace, ktlint auto-format, indent). No behavior change |
| `build` | Build config, Gradle, dependencies, wrapper |
| `chore` | Tooling, scripts, CI, hooks, formatters |
| `merge` | Merge commits (see below) |

### Allowed scopes

Match the package layout under `com.qkt.*`:

`common`, `marketdata`, `execution`, `strategy`, `broker`, `engine`, `risk`, `backtesting`, `dsl`, `app`, `research`.

Plus non-source scopes: `build`, `ci`, `docs`, `scripts`, `skill`.

If a change spans more than two scopes, drop the scope (`feat: ...`).

### Subject rules

- Imperative present tense: "add", not "added" or "adds".
- Lowercase first word.
- No trailing period.
- Maximum 70 characters.
- Describe the **what**, not the **why**. Why goes in the PR description.

### Examples

| Good | Bad |
|---|---|
| `feat(engine): add Engine orchestrator` | `Added the engine class` |
| `fix(broker): reject zero-quantity orders` | `Fix bug` |
| `refactor(common): split Clock into separate file` | `cleanup` |
| `docs: phase 2 design for event bus` | `Updated docs` |
| `build: bump assertj to 3.27.1` | `Update dependencies` |

### Merge commits

Merge commits use:
```
merge: phase <N> <short-description>
```

Example: `merge: phase 1 trading engine`. Always use `--no-ff` so the merge commit is preserved in history.

### Forbidden

- Any emoji (robot, sparkle, check-mark, anything graphical).
- `Co-Authored-By: Claude` or any other AI attribution.
- "Generated with", "Created by", or any tool-attribution footer.
- WIP commits in `main`. Squash or fix-up before merging.
- `--amend` on a published commit. Always create a new commit.

## 4. Pre-push checklist

Run before every push to a remote branch. Failing any item blocks the push.

1. `./gradlew build` — compile + run all tests + assemble. Must end with `BUILD SUCCESSFUL`.
2. `./gradlew test` — explicit test run (build also runs this, but a focused run shows individual test names).
3. `git status` — working tree must be clean. No untracked files that should have been committed; no modifications you forgot to stage.
4. `git log --oneline <base>..HEAD` — read every commit message. Every one must follow §3 conventions. No emoji, no AI footer.
5. `grep -rEn 'TODO|FIXME|XXX' src/` — any new TODOs need a github issue link or removal before push.

The repository ships a helper at `scripts/precheck.sh` that runs items 1, 2, 3 automatically. Items 4 and 5 are still on you.

## 5. Pull request requirements

Every PR has a title, a description following the template, and links to its spec/plan.

### PR title

Same format as a commit subject: `<type>(<scope>): <subject>`. Maximum 70 characters.

Plus a phase prefix in brackets at the start: `[phase 2] feat(engine): add event bus`.

### PR description template

```markdown
## Phase
Phase <N>. Spec: <link to docs/superpowers/specs/...>. Plan: <link to docs/superpowers/plans/...>.

## Summary
<1–3 sentences. The why.>

## Changes
- <bullet per meaningful change>
- ...

## Tests
- <new tests added>
- <existing tests updated>
- Local: `./gradlew build` green, <N> tests passing.

## Docs
- <new KDoc for public types/methods>
- <doc-site pages added/updated>
- Local: `./gradlew dokkaHtml` clean, `mkdocs build --strict` green.

## Backwards compatibility
<"Phase 1 only — no compat concern" OR specific notes for breaking changes>

## Out of scope
- <items deferred, with the phase or backlog where they're tracked>

## Risk
<Low / Medium / High and why. New risk-engine logic is High. A docs typo is Low.>
```

No "Generated with Claude" footers. No `🤖`.

### PR review

- At least one human reviewer (when there is a team). Solo for now: at least 24 hours between opening and merging unless trivial.
- Reviewers verify spec compliance + code quality (the same two-stage review used internally during plan execution).
- All review comments resolved before merge.

### Merge style

- `--no-ff` merge commits, never squash. Preserves the per-task commit granularity that the plan mandates.
- Merge commit message: `merge: phase <N> <feature>`.
- Delete the feature branch after merge.

## 6. Feature documentation

Every meaningful feature follows this lifecycle:

1. **Brainstorm** → produces a **design spec** at `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md`.
2. **Plan** → produces an **implementation plan** at `docs/superpowers/plans/YYYY-MM-DD-<topic>.md`.
3. **Implement** → tasks from the plan executed in order, each ending in a commit. **Every new public type, interface, top-level function, or externally-callable method MUST land with KDoc** — class-level paragraph plus one-line summary per public member. KDoc is part of the implementation, not a follow-up. See `docs/contributing/kdoc-style.md`.
4. **Merge** → into `main` via PR with the description template above.
5. **Phase changelog** → upon merging the phase, write a user-facing **phase changelog** at `docs/phases/phase-<N>-<topic>.md`. This is the developer-facing complement to the spec: the spec is forward-looking ("what we will build, why"); the changelog is backward-looking ("here's what's now available, here are all the ways to use it").

Specs document **what** and **why**. Plans document **how**, in bite-sized tasks. Phase changelogs document **how to use**, with worked examples. KDoc documents the API contract right at the source. All four are committed to git alongside the code they describe.

### Phase changelog requirements

Every phase changelog at `docs/phases/phase-<N>-<topic>.md` MUST contain:

1. **Summary** — 2–4 sentence overview of what the phase shipped and why it matters.
2. **What's new** — bullet list of every new public type, function, package, or capability. One line each. Reads like API release notes.
3. **Migration from previous phase** — if anything renamed, deprecated, or changed signature, list the before/after with a one-liner showing the textual change. Empty section is fine if no migration.
4. **Usage cookbook** — the heart of the document. Multiple worked, copy-pasteable examples covering **all the major ways** the new capabilities can be used in a strategy. Each example includes a short title, a sentence of context, and runnable Kotlin (or DSL once Phase 8 lands). Cover the common case, the power-user case, and at least one composition example showing how the new feature combines with capabilities from earlier phases. **Indicators-style phases** require examples for: simple usage, multi-symbol usage via `IndicatorMap`, composition with other indicators, custom subclass example.
5. **Testing patterns** — how the new capabilities are typically tested. Show the canonical fake or fixture and one realistic assertion.
6. **Known limitations** — what didn't ship that a reader might expect to find. Cross-link to backlog items or future phases.
7. **References** — link to the spec, the plan, and the merge commit SHA(s).

Aim for 200–500 lines per changelog. Less if the phase was small; more if it shipped a large surface area. The audience is a future contributor (or end user) who has never read the spec or plan and needs to understand what they can do with the engine today.

### `docs/` tree

```
docs/
├── superpowers/
│   ├── specs/        # design specs, one per feature
│   └── plans/        # implementation plans, one per feature
├── phases/           # per-phase changelogs (user-facing, post-merge)
│   ├── phase-1-engine-mvp.md
│   ├── phase-2-event-bus.md
│   ├── ...
├── architecture.md   # high-level system architecture (overview, refs specs)
├── phase2-backlog.md # carried-over items per phase
├── phase3-backlog.md
├── ...
```

Bug fixes do not require a spec, plan, or changelog unless the bug reveals a design flaw. In that case, the fix lands behind a refactor spec, and the affected phase changelog gets a follow-up entry.

## 7. Architecture invariants

Things that must remain true regardless of phase. Violations require a design discussion before merge.

- **Event-driven.** All system behavior is driven by events. Strategies never call brokers directly. Risk engines never call strategies. Information flows through the engine's pipeline.
- **Deterministic by default.** Any new component must be deterministic given the same inputs and seed. Random number generation is always seeded. Time access is always through `Clock`. ID generation is always through `IdGenerator`. Avoid `System.currentTimeMillis()`, `UUID.randomUUID()`, `Math.random()` in production code.
- **Single producer per type, multiple consumers.** A `MarketPriceTracker` is updated by exactly one component (the engine); read by many. A `Strategy` produces signals; the engine routes them. Don't fan-in writes.
- **Read/write split via the type system.** When a component holds mutable state, expose a read-only interface to consumers. Don't rely on convention.
- **Mock-first, real-second.** Every external integration (broker, market-data feed) ships a mock implementation before any real one. The real impl arrives behind the same interface.
- **No backwards compatibility cruft.** Phase 1 has no production users. We do not maintain compatibility shims, deprecated aliases, or transitional code paths. Refactors break things, and that's fine.
- **Strategies do not see brokers.** A strategy that imports `com.qkt.broker.*` is a design bug. The dependency arrow goes one way only — see the architecture spec for each phase.

## 8. Quant-system principles (do / don't)

### Do

- **Use `Long` milliseconds** for timestamps in Phase 1–2. Switch to `Instant` only when there's a concrete need.
- **Treat money seriously.** When P&L math arrives in Phase 3, switch to `BigDecimal` or fixed-point integers. Never accumulate `Double` over hundreds of trades — drift compounds.
- **Validate at boundaries.** Inputs from market feeds, brokers, and DSL parsers must be validated. Internal calls between trusted components do not need defensive checks.
- **Order types model real-world order types.** `Order` carries `id`, `timestamp`, `type`, `price?`. Trades carry `orderId`. These are not optional fields invented for testing — they're the audit trail.
- **Make every test reproducible.** Seed every `Random`. Inject every `Clock`. Assert on exact values, not "approximately".
- **Reject what you can't fill.** A broker that can't price an order returns `null` (rejection); it does not invent a price. A strategy that can't decide emits no signal.

### Don't

- **Don't paper over data anomalies.** A NaN price in production is a data feed bug. Crash on it (eventually) — don't silently filter it.
- **Don't use floating-point equality without tolerance** in production logic. Tests can use exact equality because we control the inputs.
- **Don't simulate fills with random latency or slippage.** The mock broker fills at the latest tracker price, period. Slippage models go in a separate, opt-in component.
- **Don't model time zones in Phase 1–4.** Everything is UTC milliseconds. Time-zone-aware reporting is a presentation-layer concern.
- **Don't invent symbol formats.** Use whatever string the data feed produces. A symbol registry is a Phase 5+ concern.
- **Don't introduce premature concurrency.** Phase 1 is single-threaded. Adding coroutines, channels, or threads is a phased decision, not a per-feature one.

## 9. DSL principles (deferred to Phase 5; recorded so we don't drift)

- The DSL exists to compile into the same `Strategy` objects we hand-write today. There is no separate runtime for DSL strategies.
- Internal Kotlin DSL first (`strategy { whenCondition { ema(9) > ema(21) }; buy("XAUUSD", 1.0) }`). External SQL-like parser only after the internal DSL stabilizes.
- The DSL should produce strategies that are deterministic and testable in the same way hand-written ones are.
- A DSL feature that doesn't have an equivalent hand-written form does not belong in the DSL.
- Performance of DSL-generated strategies must be within an order of magnitude of hand-written ones; document any divergence.

## 10. Kotlin idioms (do / don't)

### Do

- **`data class` for value types.** `Tick`, `Order`, `Trade`, `Signal.Buy`. Free `equals`/`hashCode`/`copy`/`toString`.
- **`sealed class`/`sealed interface` for closed type hierarchies.** Compiler enforces exhaustive `when`.
- **`val` over `var`.** Mutable state is tolerated only where it models real mutable state (counters, caches, `FixedClock.time`).
- **Top-level functions and properties.** No `class StringUtils { ... }` ceremony — just put a function at file top level.
- **Nullable types over `Optional`.** `Tick?` not `Optional<Tick>`. Kotlin's null safety is the language feature; use it.
- **Elvis operator** for null handling: `value ?: return null`, `value ?: error("msg")`, `value ?: defaultValue`.
- **Extension functions for transformations.** `Signal.toOrder(...)` reads better than `OrderFactory.fromSignal(signal, ...)`.
- **String templates** over concatenation: `"$prefix-${counter++}"`.
- **Single-expression functions** where the body is one expression: `override fun now(): Long = time`.
- **Default parameter values** over overloaded constructors: `class Foo(val x: Int = 0)`.
- **`require(...)` for argument validation, `error(...)` for invariant violations.** Both throw with a clear message.
- **`object : Interface { ... }`** for one-off interface impls in tests. No mocking framework needed.

### Don't

- **Don't write `public`** on declarations — it's the default, redundant.
- **Don't use `!!`** in production code. Acceptable in tests after a deliberate non-null assertion.
- **Don't import `java.util.*`** when there's a Kotlin stdlib equivalent. Use `kotlin.random.Random`, not `java.util.Random`. Use `mutableMapOf<...>()`, not `HashMap<...>()` directly.
- **Don't add Java-style getters/setters** on Kotlin properties. `val x: Int` is the property; access it as `instance.x`.
- **Don't use semicolons.** Kotlin doesn't need them.
- **Don't reach for inheritance.** Composition + interfaces is the default. `open class` requires justification.
- **Do write KDoc on public API.** Every new public type, interface, top-level function, and externally-callable method gets KDoc (class-level paragraph + one-line summary per public member). Dokka renders it into the API reference site. See `docs/contributing/kdoc-style.md` for the full style guide. The dividing line: would someone outside this package reasonably call this? If yes — KDoc it. Internal helpers and private classes do not need KDoc.
- **Don't catch `Exception`** as a way to "be safe." Catch the specific exception you handle. Let everything else propagate.

## 11. Testing standards

- **Use real types.** No Mockito, no MockK, no test-doubles libraries. Anonymous objects (`object : Strategy { ... }`) and capture lists (`val captured = mutableListOf<Trade>()`) are sufficient.
- **JUnit 5 + AssertJ.** No other frameworks until justified.
- **Test names are sentences.** Backtick-quoted: `\`fills MARKET order at tracker last price\`()`. Read like specifications.
- **One assertion per behavior, multiple assertions per setup.** Don't compress unrelated checks into one test; don't blow up one behavior into ten micro-tests.
- **Tests live in `src/test/kotlin/com/qkt/<package>/...`** mirroring the production tree.
- **Every production class with behavior has a dedicated test class.** Trivial value types (data classes, enums) are exercised indirectly.
- **Tests are deterministic.** No `Thread.sleep`, no time-dependent assertions without `FixedClock`.
- **No tests of mocked behavior.** A test that asserts "we called `broker.execute` with this Order" instead of "the order ended up at the broker" is testing implementation, not behavior.
- **Failing tests are bugs.** Fix them, escalate them, but never delete them or `@Disabled` them without an issue link.

## 12. Code style

- **File size: aim for under 150 lines per source file.** Tests may exceed by ~10% if the alternative is splitting a tightly coupled test class. Anything over 200 lines needs a refactor.
- **One top-level concept per file** — exception: tightly coupled types like `MarketPriceProvider` (interface) + `MarketPriceTracker` (impl), or a sealed class with its variants.
- **File names match the primary type.** `Order.kt` for `data class Order`. Lowercase package directories.
- **Imports.** No wildcard imports (`import java.util.*`). Group: stdlib, then third-party, then project. Most IDEs handle this automatically.
- **Naming.** Per CLAUDE.md: names describe **what**, never how or when. No `LegacyParser`, no `NewAPI`, no `EnhancedHandler`. If you can't find a clean name, the abstraction is wrong.
- **Comments.** Default to none. Only when the **why** is non-obvious — a workaround, a subtle invariant, a constraint that won't survive a casual reader. Never comment to describe what the next line of code does.
- **No dead code.** Unused imports, unused functions, commented-out blocks all get deleted before commit. `git` remembers; the file shouldn't.

## 13. Living-document protocol

When this skill is wrong or incomplete:

1. Edit it in the same PR as the code change that motivated the edit.
2. Skill commit message: `docs(skill): <what changed>`. Examples: `docs(skill): add coroutine guidance for phase 2`, `docs(skill): tighten file-size limit to 200`.
3. Don't grow this file with placeholder sections. Add a section when there's actual content.
4. When a section becomes too long for one read, split it to a sub-document under `.claude/skills/qkt/` and link from here.
5. Disagreements about conventions get resolved in PR review — once merged, the skill is the answer.
